Тест‑кейсы, статусы кейсов и тест‑сьюты

Тест-кейс — это минимальная единица тестирования, представляющая собой formalized описание действий, условий и входных данных, необходимых для проверки конкретной функции или характеристики тестируемого объекта, а также ожидаемого результата.

Основная цель тест-кейса: Обнаружить разницу между ФАКТИЧЕСКИМ поведением программы и ОЖИДАЕМЫМ, описанным в кейсе.

Структура тест-кейса (Атрибуты)

  1. ID / Уникальный идентификатор: Уникальный номер или код тест-кейса (например, TC-APP-LOGIN-01).
  2. Название / Заголовок (Title): Краткое и понятное описание того, что проверяет этот кейс (например, "Успешный вход в систему с валидными данными").
  3. Предусловия (Preconditions): Условия, которые должны быть выполнены ДО начала выполнения теста (например, "Пользователь зарегистрирован в системе и находится на странице логина").
  4. Шаги (Steps): Последовательность четких и однозначных действий, которые выполняет тестировщик.
      5. Формат: "1. Ввести логин в поле 'Username'. 2. Ввести пароль в поле 'Password'. 3. Нажать кнопку 'Login'."
  5. Ожидаемый результат (Expected Result): Описание того, как система должна повести себя после выполнения всех шагов.
      6. o Пример: "Происходит перенаправление на главную страницу аккаунта. Отображается приветствие: 'Welcome, [Username]'."
  6. Постусловия (Postconditions): Действия, которые возвращают систему в исходное состояние после выполнения теста (например, "Разлогинить пользователя").
  7. Приоритет (Priority): Важность кейса с точки зрения бизнеса.
    • P1 (High): Критически важные кейсы, проверяющие основной функционал. Выполняются в первую очередь.
    • P2 (Medium): Важные кейсы, но не блокирующие работу основных функций.
    • P3 (Low): Проверка второстепенного функционала, "счастливый путь" в нечастых сценариях.
  8. Серьезность (Severity): Ожидаемое влияние дефекта, если этот кейс упадет.
    • S1 (Blocker): Критический баг, блокирующий дальнейшее тестирование.
    • S2 (Critical): Основная функция не работает.
    • S3 (Major): Функция работает неверно.
    • S4 (Minor/Trivial): Незначительная проблема, не влияющая на основную функциональность (например, опечатка в тексте).

Статусы выполнения тест-кейса

В процессе тестирования тест-кейс может иметь следующие статусы:

  • Passed / PASS: Пройден. Фактический результат совпал с ожидаемым.
  • Failed / FAIL: Провален. Фактический результат НЕ совпал с ожидаемым. Обнаружен дефект.
  • Blocked / BLOCKED: Заблокирован. Невозможно выполнить проверку из-за критической ошибки в другом месте или отсутствия доступа.
  • Skipped / SKIPPED: Пропущен. Кейс не был выполнен намеренно.
  • Not Run / NOT EXECUTED: Не запущен. Статус по умолчанию. Кейс еще не начали выполнять.

Что такое Тест-сьют (Test Suite)?

Тест-сьют (Набор тестов) — это совокупность тест-кейсов, объединенных общей логикой или целью. Цели создания тест-сьютов:

Цели создания тест-сьютов:

  1. Организация: Группировка кейсов по функциональным модулям.
  2. Выполнение по цели: Создание наборов для конкретного типа тестирования.
    • Smoke Suite (Набор "дымового" тестирования): Минимальный набор кейсов для проверки работоспособности основных функций после сборки.
    • Regression Suite (Набор регрессионного тестирования):Большой набор кейсов, покрывающий основной функционал, чтобы убедиться, что новые изменения ничего не сломали.
    • Sanity Suite: Похож на smoke, но проводится после исправления багов, чтобы проверить, что конкретные исправления работают.
  3. Автоматизация: Группировка кейсов для автоматического запуска.

Жизненный цикл тест-кейса

  1. Создание (Draft): Аналитик/тестировщик создает кейс на основе требований, используя техники тест-дизайна.
  2. Ревью (Review): Кейс проверяют коллеги (другой тестировщик, разработчик, аналитик) на предмет корректности, полноты и отсутствия ошибок.
  3. Доработка (Rework): Автор вносит правки по комментариям ревьюера.
  4. Утверждение (Approved): Кейс готов к использованию.
  5. Выполнение (Execution): Тестировщик запускает кейс и присваивает ему один из статусов (Pass, Fail, etc.).
  6. Обновление/Снятие с эксплуатации (Update/Retire)Если функционал изменился, кейс необходимо обновить. Если функционал удален — кейс архивируется.

Вывод

Тест-кейсы — это живой артефакт. Они требуют постоянного поддержания в актуальном состоянии, так как продукт постоянно развивается.
